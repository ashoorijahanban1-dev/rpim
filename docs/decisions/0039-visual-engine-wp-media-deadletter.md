# ADR 0039 — Visual engine, WordPress media two-step, time-based dead-letter (M21)

**Status:** accepted (2026-07-19)

## Context

The publish pipeline could render HTML templates (M12) but not generate
images; WordPress photo jobs were a deliberate stub retrying forever; and a
permanently broken job had no exit from the queue. Pentarchy design §1 0016
+ §3.2 + §4.4, with the adversarial-review hardening baked in.

## Decisions

### Image generation (gateway)
- `IMAGE_PROVIDERS` mirrors the M17 text registry: `fake | openai
  (images/generations, DALL-E 3 / gpt-image-1) | gemini (Imagen predict)` —
  official APIs only (rule 5; Midjourney stays rejected). Chain env
  `MODEL_IMG` + `MODEL_IMG_FALLBACKS`; keys ride headers only (rule 4).
- `POST /image` books ledger `task="image", units=1`,
  `IMAGE_PRICES.get(model, 0.0)` (unknown models cost 0 — the PRICES
  convention). **The idempotency cache stores a lightweight receipt
  (sha256/provider/model/cost), never image bytes** — 512 cached multi-MB
  b64 blobs would OOM the single US VPS. A cached replay answers
  `{cached: true, ...}` without bytes; core-api's request_id is the
  PRE-CREATED asset id, so a network duplicate can't double-charge while a
  fresh attempt is always a fresh key (rule 8).

### Media assets (core-api)
- `media_assets` (migration **0016**, chained 0015 → 0019 → 0016 per the
  ADR 0038 numbering precedent): metadata only, bytes under
  `MEDIA_STORAGE_DIR` addressed by sha256; `UNIQUE(tenant_id, sha256)`
  dedupes content (rule 8); deterministic Persian SEO `alt_text` carries
  the subject.
- **Rule 1 covers images**: assets are born `draft`; `POST
  /studio/media/{id}/approve` (editor+) → `approved`; compile
  (`create_job`) resolves `media_asset_id` TENANT-SCOPED (foreign id = 404,
  rule 6), 409s unless approved, then marks `attached`; dispatch re-verifies
  both (defense in depth).
- Template publish rows keep the exact pre-M21 `image_spec` shape
  (`{template, size}`) — only generated rows carry
  `{kind: "generated", media_asset_id}`; old prod rows stay uniform.

### WordPress two-step (closing the M7c stub)
- Stage 1 uploads to `wp-json/wp/v2/media` (multipart, filename, alt_text)
  and the `wp_media_id` receipt **commits before stage 2** — a crash
  between stages resumes at stage 2: no orphan media, no double upload
  (rule 8, tested with an injected mid-flight crash).
- Stage 2 posts with `featured_media`. Renderer outputs get an asset row
  too (`kind=rendered`, born `approved` — rendered from an approved
  draft's frozen text), so template WP photos share the same receipt flow.
- Direct `send_photo("wordpress", ...)` keeps raising transiently — the
  engine path is the only two-stage orchestrator (it owns the session).

### Time-based dead-letter (replacing infinite retry)
- First `ChannelSendError` stamps `publish_jobs.first_failed_at`
  (app-TZ clock, ADR 0032); success clears it; silence/kill-blocked passes
  never touch it (governance is not failure).
- Age > `MAX_PUBLISH_RETRY_HOURS` (default **24** — "assume the tunnel
  WILL drop mid-job") → status `stalled`, out of the loop. Exit:
  `POST /publish/jobs/{id}/requeue` (editor+, 409 unless stalled) plus the
  stalled chip + requeue button on the publish page; `/reports/monthly`
  counts stalled.

### Export
- `export_version` → **3**: `media_assets` metadata (never bytes).

## Consequences

- Going live: set `MODEL_IMG` + provider key on the us leg,
  `IMAGE_MODE=remote` + `MEDIA_STORAGE_DIR` volume on the iran leg.
- Persian glyphs stay OUT of generated pixels (renderer constraint,
  templates.py): generated images serve as visuals; Persian text ships as
  caption/post body. Compositing text-over-generated-background is a
  future renderer slice.
- Telegram photo passthrough (M12b) carries generated bytes cross-leg
  unchanged — job_id remains the idempotency key.
